# 共享应用

共享应用是 Olares 社区应用中的一个特殊类别，由管理员统一部署和管理，集群中的其他成员无需自行安装即可使用。

本文介绍共享应用的基本概念、Olares 1.12.6 引入新架构的原因，以及如何从旧版 V2 共享应用迁移到新版 V3 架构。

## 什么是共享应用？

共享应用为 Olares 集群中的所有用户提供共享的资源或服务。与每位成员各自安装实例不同，管理员只需安装并管理一个实例，所有人即可使用。

共享应用的主要特点：

- **集中管理**：只有管理员可以安装、升级、暂停、恢复和卸载共享应用，普通成员只需使用。
- **易于识别**：在 Olares 应用市场中，共享应用通常带有 **Shared**、**共享版**、**共享应用** 或 <i class="material-symbols-outlined">group</i> 等标识。
- **灵活访问**：根据应用类型，成员可以直接从启动台打开，或将第三方客户端连接到共享 API 入口。
- **数据隔离**：所有共享应用都遵循统一的访问地址格式：

  ```text
  https://<应用 ID>.<用户名>.<平台域名>
  ```

  每位成员通过自己的用户名访问同一共享服务，系统会自动隔离各成员的数据。

共享应用分为两种形态：

- **无界面的后端服务**：仅提供 API 服务，没有图形界面。任何兼容的客户端都可以调用。例如 **Ollama Engine Base** 等引擎基座应用。
- **自带用户界面的完整应用**：同时包含后端服务和 Web 用户界面。成员可直接从启动台打开使用。例如 **Dify 共享版** 和 **ComfyUI 共享版**。

## 为什么要引入 V3 架构？

在 V1.12.6 之前，共享应用被拆分为服务端组件和客户端访问点。这种设计存在一个问题：服务端与这些访问点紧密耦合，卸载其中一个可能导致共享服务端无法访问或成为“孤儿服务”，从而影响所有用户。

V3 架构则用统一的共享服务端取代了原来的用户端/服务端拆分：

| V2 架构 | V3 架构 |
|:--------|:--------|
| 服务端 + 客户端访问点 | 统一的共享服务端 |
| 卸载某个访问点可能破坏服务端 | 服务端生命周期独立于任何访问点 |
| 访问地址和格式不统一 | 所有应用使用统一的地址格式 |
| 用户端和服务端分开管理 | 管理员只需管理一个共享服务 |

V3 架构的好处包括：

- **避免孤儿服务**：卸载客户端访问点不会再影响共享服务端。
- **管理更简单**：管理员只需管理一个共享应用，而无需协调用户端和服务端组件。
- **访问更一致**：所有共享应用使用相同的 URL 模式，方便客户端配置和书签管理。
- **数据隔离更清晰**：成员通过自己的用户名访问共享服务，系统会自动隔离数据。

## 升级到 1.12.6 后，V2 共享应用会怎样？

升级到 Olares V1.12.6 后，已安装的 V2 共享应用**不会被自动删除**。你仍可以启动、停止、暂停和恢复它们，但无法直接升级到 V3。

:::warning 数据不会自动迁移
V2 共享应用的数据**不会**自动迁移到 V3 版本。要使用新版本，需先卸载 V2 应用、安装 V3 应用，再根据应用类型迁移数据。详情请参阅[将共享应用从 V2 迁移到 V3](#将共享应用从-v2-迁移到-v3)。
:::

## 将共享应用从 V2 迁移到 V3

不同应用需要不同的迁移路径。请根据你要迁移的应用，选择对应的路径。

### 路径一：卸载 V2、安装 V3，系统自动迁移数据

以下情况使用该路径：应用支持自动数据迁移。

**属于该路径的应用：**
- **ComfyUI 共享版**

**操作步骤：**
1. 卸载 V2 共享应用。
2. 安装 V3 共享应用。
3. 系统会自动迁移你的数据。

### 路径二：卸载 V2，然后安装 V3

以下情况使用该路径：应用没有用户创建的数据需要迁移。

**属于该路径的应用：**
- **模型类应用**，如 Qwen3-Coder 30B (Ollama) 和 Gemma3 27B (vLLM)。
- **其他无重要数据的应用**，如 Falco、MTranServer 等。

**操作步骤：**
1. 卸载 V2 共享应用。
2. 安装 V3 共享应用。
3. 对于模型类应用，在引擎基座应用上重新部署该模型，并重新配置客户端。

### 路径三：备份、卸载 V2、安装 V3、然后恢复数据

以下情况使用该路径：应用存储了用户创建的数据或设置，必须手动迁移。

**属于该路径的应用：**
- **Dify 共享版** — 应用、知识库、Agent 配置和设置。
- **OnlyOffice** — 文档和应用设置。
- **SearXNG** — 搜索偏好与配置。
- **Xinference** — 已部署的模型和服务设置。

**操作步骤：**
1. **备份 V2 应用数据**

   打开**文件管理器**，进入 **Application > Data > `<应用名称>`**。下载或复制你希望保留的文件、配置或数据。

2. **卸载 V2 共享应用**

   在**应用市场**或**启动台**中卸载 V2 版本，并确保勾选**同时卸载共享服务（影响所有用户）**。这会彻底移除旧服务并释放资源。

3. **安装 V3 版本**

   在应用市场中找到该共享应用的 V3 版本并安装。此操作必须由管理员执行。

4. **迁移或重新配置数据**

   将备份的数据复制到 V3 应用的数据目录中，或按照该应用自身的迁移说明重新配置。

5. **更新访问地址**

   V3 共享应用使用统一地址格式 `https://<应用 ID>.<用户名>.<平台域名>`。请将书签或客户端配置中指向旧版 V2 地址的链接更新为新的 V3 地址。

### 路径四：从 Ollama 迁移到引擎基座

以下情况使用该路径：你从独立的 Ollama v2 应用迁移过来。

原有的独立 **Ollama** 共享应用已被 **引擎基座** 架构取代。在旧架构下，每个基于 Ollama 的模型都是独立的共享应用，推理引擎和模型打包在一起，造成了引擎重复和维护负担。在新架构下，Olares 只维护少量引擎基座应用 —— **Ollama Engine Base**、**vLLM Engine Base**、**SGLang Engine Base** 和 **llama.cpp Engine Base** —— 你根据需要在这些基座上创建模型实例。

**操作步骤：**
1. 卸载 Ollama v2 应用。
2. 选择你需要的引擎基座应用。
3. 创建所需模型的实例。
4. 在模型控制台中获取模型服务的 API 地址，并更新客户端中的地址。

## 常见问题

### V2 共享应用还会继续更新吗？

不会。V2 共享应用不会再收到新版本更新。如需新功能、修复或改进后的 V3 架构，必须先卸载 V2 版本，再安装 V3 版本。

### Ollama 应用去哪儿了？

原有的独立 **Ollama** 共享应用已被 **引擎基座** 架构取代。

在旧架构下，Olares 主要通过 Market 中的独立共享模型应用来提供模型，例如 Qwen3.5 35B A3B UD-Q4 (Ollama) 和 Qwen3 30B (vLLM)。每个应用都把推理引擎和模型打包在一起，带来了两个问题：
- **引擎重复**：同时运行多个模型意味着要运行多份相同的引擎，浪费资源。
- **维护负担重**：每新增一个模型，Olares 都需要打包并发布为一个独立的共享应用，难以扩展，也拖慢了新模型的发布速度。

在新架构下，Olares 只维护少量引擎基座应用：**Ollama Engine Base**、**vLLM Engine Base**、**SGLang Engine Base** 和 **llama.cpp Engine Base**。你选择需要的引擎基座，然后创建所需的模型实例。每个模型实例作为独立的共享服务运行，并在启动台上生成一个单独入口。

由于引擎可以共享，你可以在一个引擎基座上部署多个模型，而无需每次安装新的引擎。你还可以为不同模型选择最合适的引擎：例如本地 CPU/GPU 推理用 Ollama、高并发服务用 vLLM 或 SGLang、边缘部署用 llama.cpp。
