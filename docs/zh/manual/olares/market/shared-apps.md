# 共享应用

共享应用是 Olares 社区应用中的一个特殊类别，由管理员统一部署和管理，集群中的其他成员无需自行安装即可使用。

本文介绍共享应用的基本概念、Olares 1.12.6 引入新架构的原因，以及如何从旧版 v2 共享应用迁移到新版 v3 架构。

## 什么是共享应用

共享应用为 Olares 集群中的所有用户提供共享的资源或服务。与每位成员各自安装实例不同，管理员只需安装并管理一个实例，所有人即可使用。

共享应用的主要特点：

- **集中管理**：只有管理员可以安装、升级、暂停、恢复和卸载共享应用，普通成员只需使用。
- **易于识别**：在 Olares 应用市场中，共享应用通常带有 **Shared**、**共享版**、**共享应用** 或 <i class="material-symbols-outlined">group</i> 等标识。
- **灵活访问**：访问方式取决于应用类型。

    - **无界面的后端服务**：仅提供 API 服务，没有图形界面。例如，在引擎基座上创建的模型实例会在模型控制台中暴露一个 **Base URL**，成员将该地址粘贴到 Open WebUI 或 LobeChat 等客户端中使用。

    - **自带用户界面的完整应用**：同时包含后端服务和 Web 用户界面。成员可直接从启动台打开使用。例如 **Dify 共享版** 和 **ComfyUI 共享版**。

- **统一 HTTPS 地址**：所有共享应用都使用相同的 URL 格式，即：`https://<应用 ID>.<用户名>.<平台域名>`，每位成员通过自己的用户名访问同一共享应用。

## 为什么引入 v3 架构

在 v1.12.6 之前，共享应用被拆分为服务端组件和客户端访问点。这种设计存在一个问题：服务端与这些访问点紧密耦合，卸载其中一个可能导致共享服务端无法访问，从而影响所有用户。

v3 架构则用统一的共享服务端取代了原来的用户端/服务端拆分：

| v2 架构 | v3 架构 |
|:--------|:--------|
| 服务端 + 客户端访问点 | 统一的共享服务端 |
| 卸载某个访问点可能破坏服务端 | 服务端生命周期独立于任何访问点 |
| 访问地址和格式不统一 | 所有应用使用统一的地址格式 |
| 用户端和服务端分开管理 | 管理员只需管理一个共享服务 |

v3 架构的好处包括：

- **管理更简单**：管理员只需管理一个共享应用，而无需协调用户端和服务端组件。
- **避免孤儿服务**：卸载客户端访问点不会再影响共享服务端。
- **访问更个性化**：所有共享应用使用相同的 HTTPS URL 模式，每位成员通过自己的用户名访问，拥有独立的地址。

## v2 共享应用生命周期

升级到 Olares v1.12.6 后，你仍可以像以前一样启动、使用、暂停和恢复它们，但无法直接升级到 v3。

:::warning 数据不会自动迁移
v2 共享应用的数据不会自动迁移到 v3 版本。要使用新版本，需先卸载 v2 应用、安装 v3 应用，再根据应用类型迁移数据。详情见[将共享应用从 v2 迁移到 v3](#将共享应用从-v2-迁移到-v3)。
:::

## 从 Ollama 到引擎基座

原有的独立 Ollama 共享应用已被新的引擎基座架构取代。

在旧架构下，Olares 主要通过独立的共享模型应用来提供模型，例如 Qwen3.5 35B A3B UD-Q4 (Ollama)。每个应用都把模型和自己的推理引擎打包在一起。这种设计存在两个主要问题：

- **调度冲突**：独立的 Ollama 应用与 Olares 的 GPU 时间分片管理不兼容，同时运行多个模型时可能会在 GPU 上产生冲突。
- **强耦合**：由于引擎和模型打包在同一个应用中，更新引擎或新增模型都需要为每个模型重新发布一个应用，导致上架新模型的速度变慢。

新的引擎基座架构保留了相同的基本运行方式，每个模型实例仍各自运行自己的引擎，但把引擎层抽象成了可复用的基座应用：
- Olares 只维护少量引擎基座应用：**Ollama Engine Base**、**vLLM Engine Base**、**SGLang Engine Base** 和 **llama.cpp Engine Base**。
- 你选择需要的基座，然后创建所需的模型实例。每个实例作为独立的共享服务运行，并在启动台上生成一个单独入口。

这带来了以下优势：
- **兼容 GPU 时间分片**：模型实例遵循 Olares 的调度机制，因此你可以同时运行多个模型，而不会出现旧版独立 Ollama 应用那样的冲突。
- **在应用市场中集中管理**：引擎通过基座应用统一维护。当 Olares 更新某个基座后，你可以把基于它克隆的实例升级到新的引擎版本。
- **配置更灵活**：你可以自行设置每个实例的模型来源、参数和能力，并为不同模型选择最合适的引擎。例如，本地快速推理用 Ollama、高并发服务用 vLLM 或 SGLang、边缘部署用 llama.cpp。

## 将共享应用从 v2 迁移到 v3

不同应用需要不同的迁移方法。请根据你要迁移的应用，选择对应的方法。

### 方法一：卸载 v2、安装 v3，系统自动迁移数据

以下情况使用该方法：应用支持自动数据迁移。

**属于该方法的应用：**
- **ComfyUI 共享版**

**操作步骤：**
1. 卸载 v2 共享应用。
2. 安装 v3 共享应用。
3. 系统会自动迁移你的数据。

:::tip ComfyUI 数据位置有变化
在 v2 中，ComfyUI 的主程序存放在`外部设备`目录下。在 v3 中，主程序迁移到`数据`目录下，模型文件迁移到 `公共`目录下。如果你需要直接访问这些文件，请注意迁移后的新位置。详情见 [ComfyUI 共享版迁移指南](#)。<!-- TODO: 相关子页面发布后，替换为实际链接 -->
:::

### 方法二：卸载 v2，然后安装 v3

以下情况使用该方法：应用没有用户创建的数据需要迁移。

**属于该方法的应用：**
- **模型类应用**，如 Qwen3-Coder 30B (Ollama) 和 Gemma3 27B (vLLM)。
- **其他无重要数据的应用**，如 Falco、MTranServer 等。

**操作步骤：**
1. 卸载 v2 共享应用。
2. 安装 v3 共享应用。
3. 对于模型类应用，在引擎基座应用上重新部署该模型，并重新配置客户端。

### 方法三：备份、卸载 v2、安装 v3、然后恢复数据

以下情况使用该方法：应用存储了用户创建的数据或设置，必须手动迁移。

**属于该方法的应用：**
- **Dify 共享版** — 应用、知识库、Agent 配置和设置。
- **OnlyOffice** — 文档和应用设置。
- **SearXNG** — 搜索偏好与配置。
- **Xinference** — 已部署的模型和服务设置。

<!-- TODO: Dify 共享版、OnlyOffice、SearXNG、Xinference 的单独迁移教程尚未在 use cases 中编写。目前下方的通用步骤覆盖所有这些应用。等各应用的 use-case 页面发布后，再把每个应用链接到各自的教程。 -->

**操作步骤：**
1. **备份 v2 应用数据**

   打开**文件管理器**，进入 **Application > Data > `<应用名称>`**。下载或复制你希望保留的文件、配置或数据。

2. **卸载 v2 共享应用**

   在**应用市场**或**启动台**中卸载 v2 版本，并确保勾选**同时卸载共享服务（影响所有用户）**。这会彻底移除旧服务并释放资源。

3. **安装 v3 版本**

   在应用市场中找到该共享应用的 v3 版本并安装。此操作必须由管理员执行。

4. **迁移或重新配置数据**

   将备份的数据复制到 v3 应用的数据目录中，或按照该应用自身的迁移说明重新配置。

5. **更新访问地址**

   v3 共享应用使用统一地址格式 `https://<应用 ID>.<用户名>.<平台域名>`。请将书签或客户端配置中指向旧版 v2 地址的链接更新为新的 v3 地址。

### 方法四：从 Ollama 迁移到引擎基座

以下情况使用该方法：你从独立的 Ollama v2 应用迁移过来。有关独立 Ollama 应用为何被引擎基座取代的背景，见[从 Ollama 到引擎基座](#从-ollama-到引擎基座)。

**操作步骤：**
1. 卸载 Ollama v2 应用。
2. 选择你需要的引擎基座应用。
3. 创建所需模型的实例。
4. 在模型控制台中获取模型服务的 API 地址，并更新客户端中的地址。

## 常见问题

### 关于共享入口

系统中仍然存在**共享入口**，但你不要使用它来访问共享应用。该地址不区分用户，今后仅保留给系统级内部集成使用，例如需要统一地址的预配置 Agent-to-Agent API 调用。

你自己访问共享应用时，可直接从启动台打开；连接客户端时，请使用模型控制台中显示的 Base URL。
